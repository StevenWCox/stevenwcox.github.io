Logger

global inherited sharing class Logger
DescriptionThe lws.Logger's primary responsibility is to create lws Log records. It utilizes immediate Platform Events to circumvent rollbacks that happen when an exception is thrown in a transaction. After calling lws.Logger.log, you can be assured that the log record will be committed. Batchable classes that implement Database.RaisesPlatformEvents will auto-generate log records.

An additional benefit of lws.Logger is that it can be turned on and off via a hierarchical custom setting, lws.Settings. Just adjust the Log Level for any user or profile - even in production - to start creating log records to triage problems.

In addition to the object you pass to lws.Logger.log, ApexBoost will record the context (class, method, line), current user, and stack trace.

Finally, build reports and workflow on Logs to notify stakeholders of errors occurring in the system.

Usage:

 // ERROR|scox~ MyClass.myMethod:14 simple test
 lws.Logger.log('simple test');
 

 // WARN|scox~ MyClass.myMethod:16 warning test
 lws.Logger.log(LoggingLevel.WARN, 'warning test');
 

 // ERROR|Class.MyClass.execute:16
 Database.executeBatch(new MyClass());

Logger.Cleaner

global without sharing class Cleaner extends BatchJob.Handler
DescriptionA batch process that deletes old log records. On initial installation of ApexBoost, this process is scheduled to run at 1am every day. The Log Days To Keep custom setting field specifies how many days log records will be kept. The default is 30.

Usage:

 // schedule the job to run every day at 1am
 System.schedule('Clean Log Records', '0 0 1 * * ?', new lws.BatchJob(new lws.Logger.Cleaner()));
 

 // clean old logs immediately
 new lws.Logger.Cleaner().run();

TriggerHandler

global virtual inherited sharing class TriggerHandler
DescriptionThis class provides trigger handler management with
  • A consistent and efficient pattern for trigger handler implementation.
  • A central dispatch mechanism
  • The ability to disable all triggers
  • The ability to disable a single trigger

Usage:

 // create a handler class. This class must be global
 global with sharing class Accounts extends lws.TriggerHandler {
   // override TriggerHandler methods needed for your logic...
   global override void onBeforeInsert(SObject[] records) {
     doSomethingWithAccounts();
   }
 }
 

 // hook up your trigger to the framework using TriggerHandler.handle.
 trigger Accounts on Account(before insert) {
   TriggerHandler.handle(Accounts.class);
 }

Disabling Triggers

To disable all triggers, set the ApexBoost Enable Triggers custom Setting to false. Note that this can be done for an individual user or profile.

To disable a single trigger, create a Boolean ApexBoost custom Setting with the same name as your handler class. This new setting can now be set to true to disable the handler globally or for a single user or profile.

Assert

global inherited sharing class Assert
DescriptionA suite of assertion methods meant to be used withing your Apex code, not in unit tests. Such assertions are commonly used to validate assumptions such as input parameters, object state, or return values. Note that assertions should never be used for error-handling. Instead, use lws.Assert to enforce and document design-time restrictions.

ApexBoost assertions have two primary advantages over System.assert methods:

  1. They can be caught by custom code. Unlike System.AssertException, lws.AssertException is a custom exception and can therefore be caught. This makes it possible for you to use lws.Assert within your code, and write unit tests that validate those assertions.
  2. They can be disabled with lws.Settings, allowing you to utilize assertions during development and debugging, but not incur the overhead in production.

Usage:

 public with sharing class MyClass {
   public MyClass(Id imporantId) {
     lws.Assert.areNotEqual(null, importantId, 'an importantId is required!');
     ...
   }
 }
 

 @IsTest class MyClassTest {
   @IsTest static void testConstructor() {
     Test.startTest();
       try {
         new MyClass(null);
         System.assert(false, 'a missing importantId should fail');
       } catch (lws.AssertException expected) {}
     Test.stopTest();
   }
 }

BatchJob

global inherited sharing class BatchJob implements Database.Batchable<sObject>, Database.AllowsCallouts, Database.RaisesPlatformEvents, Database.Stateful, Schedulable
DescriptionA generic class for scheduled batch operations. Instead of requiring several new classes (batchable, schedulable, test...) for each job, this design allows you to add batch jobs as nested classes within an existing Apex class. The effect is to greatly reduce the number of Apex classes and boiler-plate code.

lws.BatchJob handles all the plumbing (debug logging, exception logging, etc) and delegates the interesting work (your business logic) to your code. To use this model, create a class that extends lws.BatchJob.Handler and override start and execute.

 public class Accounts {
   public class Cleaner extends lws.BatchJob.Handler {
     public Database.QueryLocator start(Database.BatchableContext context) {
       return new lws.Query(Account.SObjectType)
         .whereOp(Account.Purge__c, '=', true)
         .locator();
     }
     public void execute(Database.BatchableContext context, SObject[] scope) {
       delete scope;
     }
   }
 }

Usage:

 // execute an account cleaner job
 new Accounts.Cleaner().run();
 

 // execute a batch with a custom batch size
 Database.executeBatch(new lws.BatchJob(new Accounts.Cleaner()), 5);
 

 // schedule the job with a custom batch size, starting in 10 minutes
 System.scheduleBatch(new lws.BatchJob(new Accounts.Cleaner(), 5), 'cleaning...', 10);
 

 // schedule the job with a custom batch size; run every day at 1pm
 System.schedule('cleaning...', '0 0 13 * * ?', new lws.BatchJob(new Accounts.Cleaner(), 5));

In addition to Batchable and Schedulable, this class implements Database.AllowsCallouts, Database.RaisesPlatformEvents, and Database.Stateful, which provide extra functionality.

Util

global inherited sharing virtual class Util
DescriptionThis class contains diagnostic methods with several advantages over their System counterparts:
  • Context (class, method, line number) is auto-generated, saving you typing and debugging time! The simplest debug statement generates a wealth of information: 

     lws.debug(); // => DEBUG|scox~ MyClass.myMethod:14

  • Methods can be essentially disabled using ApexBoost custom settings. For example, setting Debug Level to ERROR will prevent all lws.debug() calls from generating a debug entry.

In order for the debug/warning/info methods to work, you'll need to create a global class extending lws.Util. Specify this class in the ApexBoost custom settings, then override the onDebug method to write information to System.debug. 

 global with sharing class LwsDebug extends lws.Util {
   global override void onDebug(LoggingLevel level, String details) {
     System.debug(level, details);
   }
 }

Query

global inherited sharing class Query
DescriptionUsing this class, a dynamic query can be constructed piece by piece, adding object and field type checking, which avoids typos and references to invalid fields. Object read permission and FLS may also optionally be checked.

Usage:

 // get user names where City starts with "San"
 User[] users = new lws.Query(User.SObjectType)
   .field(User.Name)
   .whereOp(User.City, 'LIKE', 'San%')
   .run();
 

 // get the number of contacts in Dallas accounts
 lws.Query dallasAccounts = new lws.Query(Account.SObjectType).whereOp(Account.City, '=', 'Dallas'));
 lws.Query contacts = new lws.Query(Contact.SObjectType);
 contacts.field(Contact.Name);
 contacts.whereIn(Contact.AccountId, dallasAccounts);
 Integer contactCount = contacts.count();
 

 // get the contacts in certain accounts
 Account[] results = new lws.Query(Account.SObjectType)
   .fields(new Query().createChildQuery(Contact.SObjectType, 'Contacts'))
   .whereIn(Account.Id, accounts)
   .run();
